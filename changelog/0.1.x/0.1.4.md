# 0.1.4 — 2026-06-06

## Fixed

- **`fema_get_disaster` / `fema://disaster/{number}`** — disaster numbers above 32767 now throw `not_found` immediately instead of reaching the OpenFEMA API and returning a raw Int16 type-mismatch error. OpenFEMA's OData layer stores `disasterNumber` as Int16 (max 32767); no real FEMA disaster approaches that ceiling. ([#2](https://github.com/cyanheads/fema-mcp-server/issues/2))
- **`fema_search_disasters` / `fema_get_public_assistance` / `fema_get_housing_assistance`** — the no-results recovery hint is now embedded in `data.recovery.hint` (present in `structuredContent`) rather than in `ctx.enrich.notice()`, which only fires on the success path and is unreachable from a thrown error. ([#3](https://github.com/cyanheads/fema-mcp-server/issues/3))
- **`fema_query_dataset`** — 400 responses from the OpenFEMA OData layer are sanitized before surfacing: parser position offsets (`at 469`) and undefined error codes (`[undefined]`) are stripped; the caller receives an actionable OData guidance message instead. ([#4](https://github.com/cyanheads/fema-mcp-server/issues/4))
